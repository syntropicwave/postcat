# postcat — предложение и план разработки

> Дата: 06.07.2026. Исследования, на которых основан документ: [docs/research/postman-features.md](docs/research/postman-features.md) (функциональность Postman) и [docs/research/competitor-stacks.md](docs/research/competitor-stacks.md) (стеки конкурентов).

## 1. Позиционирование

**postcat — локальный десктопный API-клиент, в котором история запросов — главный рабочий инструмент, а не побочный лог.**

Ключевая идея: разработчик за день отправляет десятки запросов, и его реальная потребность — «найти тот запрос, который я делал в прошлый вторник, где в ответе был `order_id=...`». Ни один из существующих клиентов это не умеет:

| Клиент   | История                                                                                                                                                                      |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Postman  | Поиск только по URL и только по подгруженной части списка; тела ответов по умолчанию не сохраняются; ретеншн привязан к тарифу и логину; нет фильтров по методу/статусу/дате |
| Insomnia | Ответы хранятся в мёртвой NeDB, поиска по истории фактически нет                                                                                                             |
| Bruno    | История живёт только в рамках сессии; issues #411/#6742 с просьбами о постоянной истории с поиском открыты годами                                                            |
| Yaak     | SQLite есть, полнотекстового поиска нет                                                                                                                                      |

Второй столп позиционирования — **local-first без аккаунта**. Postman (сентябрь 2023, удаление Scratch Pad) и Insomnia (8.0, принудительный логин) синхронно обожгли пользователей — рынок это помнит, и «все данные у вас, офлайн, навсегда» — обязательное условие входа, а не фича.

Коллекции поддерживаем полноценно (включая импорт Postman Collection v2.1, cURL, OpenAPI), но UX строится от истории: «сохранить в коллекцию» — это действие над записью истории, а не наоборот.

### Что именно делает историю «качественной»

1. **Полнотекстовый поиск по всему**: URL, заголовки, тело запроса, тело ответа. Мгновенно (< 50 мс на десятках тысяч записей), офлайн, с подсветкой совпадений и сниппетами.
2. **Фильтры**: метод, код статуса (класс 2xx/4xx/… или точный), хост, дата/диапазон дат, workspace, «только с ошибками», источник (вручную / runner / из коллекции X).
3. **Ответы сохраняются всегда** (с настраиваемым лимитом размера тела), а не «выключено по умолчанию», как в Postman.
4. **Неограниченный ретеншн** локально + политика очистки по желанию пользователя (по возрасту/размеру), а не по тарифу.
5. **Действия над историей**: повторить запрос, открыть как черновик, diff двух записей (что изменилось в запросе и в ответе), сохранить в коллекцию, закрепить/пометить ярлыком.
6. **Группировка**: соседние записи с одинаковым методом+URL сворачиваются в «таймлайн одного эндпоинта» — видно, как менялся ответ между попытками.

## 2. Предложение по стеку

### Рекомендуемый стек

| Слой                  | Выбор                                           | Обоснование                                                                                                                                                                                                                                                                         |
| --------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Оболочка              | **Tauri 2**                                     | Инсталлятор < 10 МБ против 100+ у Electron, ~в 4–5 раз меньше памяти. Рынок высмеивает «тяжёлый Postman» — лёгкость это часть позиционирования. Прецеденты в нашей нише: Yaak и Hoppscotch Desktop уже на Tauri 2                                                                   |
| Backend / HTTP-движок | **Rust + reqwest (hyper)**                      | Запросы уходят из нативного процесса: нет CORS и браузерных ограничений, полный контроль TLS/mTLS/прокси/редиректов/HTTP2 и точный тайминг фаз (DNS, connect, TLS, TTFB). Ради этого Hoppscotch построил отдельный Rust-агент — у нас это бесплатно                                 |
| Хранилище             | **SQLite (rusqlite) + FTS5**                    | Одна файловая БД: история, коллекции, окружения, транзакции. FTS5 external-content таблица над url/заголовками/телами (BM25-ранжирование, snippet/highlight), trigram-токенизатор на колонке URL для подстрочного поиска. Проверено браузерами: история Chrome/Firefox — это SQLite |
| Большие тела ответов  | **Файлы на диске** (модель Yaak)                | В SQLite — метаданные и первые N КБ текста для индекса; тело целиком — файл в `app_data/responses/`. БД не распухает от гигабайтных ответов                                                                                                                                         |
| Frontend              | **React + TypeScript + Vite**                   | Максимальная экосистема, найм,知 инструментов. State: Zustand/TanStack Query                                                                                                                                                                                                        |
| Редакторы кода        | **CodeMirror 6**                                | Легче Monaco, модульный, отличная подсветка JSON/XML/GraphQL; Monaco оставить как опцию, если понадобится LSP                                                                                                                                                                       |
| Скриптовый сандбокс   | **rquickjs (QuickJS-NG)**                       | ~200 КБ, старт мгновенный, лимиты памяти/времени на контекст, никаких ambient-возможностей (fs/network появляются только если мы их дали). Индустрия сошлась там же: Bruno Safe Mode = QuickJS-in-WASM. vm2 мёртв (8 критических CVE), isolated-vm в maintenance mode               |
| Импорт форматов       | Postman Collection v2.1, cURL, OpenAPI 3.x, HAR | Открытые спецификации; v2.1 — JSON Schema на schema.postman.com                                                                                                                                                                                                                     |

### Рассмотренные альтернативы (и почему нет)

- **Electron + React** — безопасный, самый быстрый старт для JS-команды, консистентный рендеринг. Но: 150–300 МБ RAM в простое, 100+ МБ инсталлятор, и нам всё равно пришлось бы делать запросы из Node, а поиск — из better-sqlite3. Осмысленный запасной вариант, если Rust-компетенции в команде нет совсем. Проигрывает по позиционированию («ещё один тяжёлый клиент»).
- **Tantivy вместо FTS5** — лучше ранжирование и фасеты, но это второе хранилище, которое надо держать консистентным с SQLite. Апгрейд для v2, если история дорастёт до миллионов записей; не для v1.
- **deno_core (V8) вместо QuickJS** — почти полная Node-совместимость скриптов (важно для фиделити импорта Postman-скриптов), но +десятки МБ бинарника и сложность сборки. Держим как план Б, если совместимость pm.* на QuickJS окажется болезненной; либо Node-sidecar по модели Yaak.
- **Веб-версия (по модели Hoppscotch)** — CORS превращает архитектуру в «зоопарк интерцепторов» (расширение, прокси, локальный агент). Не наш путь; десктоп решает это по построению.

Главный принятый риск Tauri: рендеринг на Linux (WebKitGTK отстаёт от Chromium) и необходимость QA на трёх webview. Для UI из форм и панелей это приемлемо; Windows (WebView2) и macOS (WKWebView) зрелые.

## 3. Архитектура (эскиз)

```
┌────────────────────────── Tauri 2 shell ──────────────────────────┐
│  WebView (React + TS)                                             │
│  ├─ Request Builder (URL/params/headers/body, CodeMirror)         │
│  ├─ Response Viewer (pretty/raw/preview, поиск, метрики)          │
│  ├─ History Explorer (поиск FTS + фильтры + diff)  ← центр UX     │
│  └─ Collections / Environments sidebar                            │
│         │  tauri IPC (commands + events)                          │
│  Rust core                                                        │
│  ├─ http_engine: reqwest/hyper — исполнение запросов,             │
│  │   cookie jar, тайминги фаз, отмена, стриминг больших тел       │
│  ├─ store: rusqlite + миграции                                    │
│  │   ├─ history, requests, collections, environments, cookies     │
│  │   ├─ FTS5 (external content) + триггеры синхронизации          │
│  │   └─ blob-store: тела ответов на диске, ссылки в БД            │
│  ├─ script_engine: rquickjs — pre-request / test (pm.* subset)    │
│  └─ importers: postman v2.1 / curl / openapi / har                │
└───────────────────────────────────────────────────────────────────┘
```

Ключевые схемные решения:

- `history_entries`: id, sent_at, method, url, host, path, status, duration_ms, req_headers, req_body_ref, resp_headers, resp_body_ref, размеры, тайминги фаз, environment_id, collection_request_id (nullable — связь с «родительским» сохранённым запросом), source (manual/runner/replay).
- FTS-таблица индексирует: url, req_headers_text, req_body_text (первые N КБ), resp_body_text (первые N КБ). Отдельная trigram-колонка для URL-подстрок.
- История — **append-only**: редактирование запроса никогда не меняет прошлые записи. «Сохранённый запрос» в коллекции — это отдельная сущность, у которой есть своя цепочка истории.
- Секреты (значения переменных с типом secret) в текст индекса не попадают и в истории маскируются на этапе записи — прямой ответ на известные утечки секретов в Postman.

## 4. План разработки

Фазы упорядочены так, чтобы дифференциатор (история) появился как можно раньше, а «таблица ставок» (коллекции, auth, скрипты) — следом. Оценки — в неделях чистой разработки для 1–2 человек.

### Фаза 0 — Каркас (1–2 нед)

- Tauri 2 + React + TS + Vite скелет, CI (GitHub Actions: build win/mac/linux), линтеры (clippy, eslint), rusqlite + система миграций.
- IPC-контракт (typed commands), логирование, крэш-репорты локально.
- **Выход:** пустое окно, которое собирается в инсталляторы на трёх ОС.

### Фаза 1 — MVP: запрос → ответ → история (3–4 нед)

- Request builder: методы, URL c двусторонней синхронизацией params-таблицы, path-переменные, заголовки (autocomplete, enable/disable), body: raw (JSON/XML/text, подсветка+beautify), form-data (с файлами), urlencoded, binary.
- Отправка через reqwest: отмена, таймауты, редиректы, verify-SSL toggle, cookie jar (автоматический).
- Response viewer: pretty/raw/preview, подсветка, сворачивание узлов JSON, Ctrl+F по ответу, статус/время/размер, тайминги фаз, вкладки Headers/Cookies.
- **Каждый запрос автоматически пишется в историю целиком (включая ответ)** — с первого дня.
- Простой список истории по датам, повтор запроса из истории.
- Вкладки (мульти-запрос), горячие клавиши.
- **Выход:** ежедневно пригодный инструмент уровня «curl с UI + вечная память».

### Фаза 2 — История как продукт (3–4 нед) ← дифференциатор

- FTS5-индекс + поисковая строка: полнотекст по URL/заголовкам/телам запроса и ответа, сниппеты с подсветкой, ранжирование BM25, «мгновенный» инкрементальный поиск.
- Фильтры: метод, статус/класс статуса, хост, диапазон дат, источник; комбинируются с текстовым запросом; сохраняемые фильтры.
- Diff двух записей истории (запрос и ответ раздельно, JSON-aware).
- Группировка по эндпоинту (метод+нормализованный URL), таймлайн изменений ответа.
- Pin/label записей; политика хранения (по возрасту/объёму, по умолчанию — хранить всё); экспорт записи в cURL/HAR.
- Бенчмарк: поиск < 50 мс при 100k записей (сид-скрипт для генерации истории).
- **Выход:** главная причина выбрать postcat работает и демонстрируема.

### Фаза 3 — Коллекции, окружения, импорт (3–4 нед)

- Коллекции: папки любой вложенности, сохранённые запросы, examples (запрос+ответ), описания (Markdown), drag&drop; «сохранить из истории в коллекцию» как первичный сценарий.
- Переменные и окружения: скоупы global / environment / collection / local, `{{var}}` с автокомплитом и подсветкой, initial vs current value (current никогда не покидает машину), secret-тип с маскированием (и исключением из FTS).
- Импорт: Postman Collection v2.1 (+environments), cURL (paste в URL-бар), OpenAPI 3.x, HAR. Экспорт: v2.1, cURL.
- Опционально (решить в фазе): зеркалирование коллекций в текстовые файлы на диске для git (модель Yaak).
- **Выход:** миграция с Postman за 10 минут.

### Фаза 4 — Auth, cookies, настройки (2–3 нед)

- Auth-хелперы с наследованием от коллекции/папки: API Key, Bearer, Basic, **OAuth 2.0 (Authorization Code + PKCE, Client Credentials, refresh + менеджер токенов)**, Digest, AWS SigV4; остальные (OAuth 1.0, NTLM, Hawk, JWT-генератор) — по спросу.
- Cookie Manager (просмотр/редактирование jar по доменам).
- Сертификаты: клиентские (PFX/PEM) по маскам хостов, кастомные CA.
- Настройки: прокси (системный/кастомный + bypass), таймауты, лимит размера ответа, редиректы, темы.
- Генерация кода из запроса: cURL, fetch, python-requests, Go, C# (список расширяем).
- **Выход:** паритет с Postman по ежедневным сценариям работы с реальными API.

### Фаза 5 — Скрипты и runner (3–4 нед)

- rquickjs-сандбокс: pre-request и test-скрипты на уровне запроса/папки/коллекции; совместимое подмножество `pm.*` (pm.request/response/test/expect (chai-подобные ассерты)/variables/environment/sendRequest); лимиты памяти/времени; console с выводом в панель.
- Collection Runner: последовательный прогон, итерации, data-файлы CSV/JSON, задержки, отчёт по тестам, setNextRequest.
- CLI-раннер (тот же Rust-core, headless) для CI — аналог Newman.
- **Выход:** автоматизация и тесты; импортированные Postman-коллекции со скриптами в основном работают.

### Фаза 6 — Протоколы (по спросу, 2–4 нед каждая)

- GraphQL: introspection, schema explorer, query builder (первый приоритет — дёшево поверх HTTP).
- SSE (стриминг в панель ответа), WebSocket (сообщения — тоже в историю с поиском!), позже gRPC (tonic, reflection) и MQTT.
- Стриминговые сессии пишутся в историю как записи с вложенными сообщениями — поиск по содержимому сообщений сохраняет дифференциатор и здесь.

### Фаза 7 — Полировка и дистрибуция (параллельно с 5–6)

- Авто-апдейты (tauri-updater), подпись бинарников, winget/homebrew, лендинг с позиционированием «история, которая помнит всё».
- Производительность: холодный старт < 1 c, поиск < 50 мс, память в простое < 100 МБ — публикуемые метрики, часть маркетинга.

**Итого до «паритет + дифференциатор» (фазы 0–4): ~12–17 недель; до полной v1.0 со скриптами: ~15–21 неделя** (1–2 разработчика).

## 5. Осознанно НЕ делаем в v1

- Облачную синхронизацию и аккаунты (можно добавить позже как opt-in E2EE; см. уроки Postman/Insomnia).
- Mock-серверы, мониторинг, публикацию документации, AI-фичи — источник «бламинга» Postman за feature creep.
- Socket.IO, MQTT, SOAP-хелперы — до явного спроса.
- Плагины/расширения — API стабилизируем после v1.

## 6. Риски

| Риск                                   | Митигация                                                                                                                                                                            |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Rust-компетенция команды               | Ядро (engine+store) компактное; UI — обычный React. Если Rust — блокер, запасной стек: Electron + better-sqlite3 (FTS5 доступен и там) — дифференциатор сохраняется, теряем лёгкость |
| WebKitGTK на Linux                     | Ранний smoke-тест UI на Ubuntu в CI; избегать экзотического CSS                                                                                                                      |
| Совместимость pm.*-скриптов на QuickJS | Целимся в «совместимое подмножество», прогоняем корпус реальных Postman-коллекций; план Б — deno_core или Node-sidecar (модель Yaak)                                                 |
| Рост БД от тел ответов                 | Тела на диске, в индексе первые N КБ, настраиваемые лимиты и политика очистки                                                                                                        |
| Yaak добавит FTS раньше нас            | Скорость: фаза 2 — приоритет №1 после MVP; история у нас глубже (diff, группировка, фильтры), чем «просто поиск»                                                                     |
