# postcat — синхронизация (дизайн)

> Статус: proposal, 2026-07-06. Облако — строго **opt-in**, выключено по умолчанию. Ничего в существующем local-first поведении не меняется, пока пользователь сам не включит sync.

## Принципы

1. **Local-first остаётся источником правды.** SQLite на машине — основной store; sync — это надстройка, а не зависимость. Офлайн всё работает как раньше.
2. **E2E по построению.** Сервер хранит только зашифрованные блобы; ключ выводится из пароля на клиенте (Argon2id) и **никогда не покидает устройство**. Наш хостинг и self-hosted инстанс — одинаковый «тупой» зашифрованный store. Даже админ self-host не читает содержимое.
3. **Self-host из коробки.** Сервер — один маленький Rust-бинарь (axum + SQLite), который компания запускает у себя. В настройках postcat указывается URL сервера. Никакого Postgres/Redis для старта.
4. **Секреты и история — под явным контролем.** По умолчанию синхронизируются коллекции и окружения (initial values). `current value` (локальные, в т.ч. секретные) и история — **отдельные тумблеры**, выключены по умолчанию.

## Что синхронизируется

| Данные | По умолчанию | Примечание |
|---|---|---|
| Коллекции (папки, запросы, скрипты, auth-конфиги) | ✅ | ядро sync |
| Окружения + переменные (initial value) | ✅ | |
| `current value` переменных (локальные оверрайды, секреты) | ❌ opt-in | наша модель initial/current уже разделяет это |
| История запросов/ответов | ❌ opt-in | большой объём + чувствительно; отдельный тумблер |
| Настройки приложения, cookies | ❌ | остаются локальными |

## Архитектура

```
   postcat (клиент)                      postcat-sync (self-host бинарь)
 ┌────────────────────┐                ┌───────────────────────────────┐
 │ SQLite (правда)    │                │ axum + SQLite                 │
 │  + updated_at,     │  HTTPS         │  users(email, auth_verifier)  │
 │    deleted (tomb)  │ ───────────▶   │  blobs(user, item_id, rev,    │
 │  sync engine       │  push/pull     │        ciphertext, updated_at)│
 │  crypto (Argon2id, │ ◀───────────   │  — видит только шифртекст     │
 │   XChaCha20-Poly)  │                └───────────────────────────────┘
 └────────────────────┘
```

- **Change tracking:** к синхронизируемым таблицам добавляются `updated_at` и `deleted` (tombstone). Каждое изменение проставляет `updated_at`; удаление ставит tombstone (чтобы удаление доехало до других устройств).
- **Единица синхронизации** — «item» (коллекция целиком или отдельный запрос — решим по гранулярности; старт: сущность = строка коллекции/окружения). Клиент шифрует JSON сущности → `ciphertext`, отправляет с `rev` (ревизия) и `updated_at`.
- **Конфликты:** per-item last-writer-wins по `updated_at` в v1; в UI показываем «изменено на другом устройстве», без слияния (слияние — позже, при спросе). Git-модель (см. ниже) даёт нормальный merge тем, кому нужно.
- **Crypto:** пароль → Argon2id → master key. Контент шифруется XChaCha20-Poly1305 (per-item nonce). Сервер получает email + верификатор аутентификации (тоже из пароля, отдельная ветка KDF), не мастер-ключ.

## Два уровня (не взаимоисключающие)

**Уровень 0 — Git-mirror (без сервера вообще).**
Зеркалирование коллекций/окружений в текстовые файлы на диске (наш формат или Postman v2.1). Пользователь коммитит в свой git. Даёт бэкап, историю версий, шаринг и merge через инфраструктуру, которой уже доверяют. Ноль нашего кода на сервере. Это фундамент change-tracking'а и полезно само по себе.

**Уровень 1 — E2E sync-сервер (opt-in).**
Тот же change-tracking + шифрование + `postcat-sync` бинарь. Автоматическая синхронизация между устройствами, self-hostable.

## План постройки (по фазам, как обычно)

- **S0. Change tracking + Git-mirror.** `updated_at`/`deleted` в схеме, экспорт/импорт дерева в файлы, «Sync folder» в настройках, кнопки pull/push из файлов. Ценность сразу, сервер не нужен.
- **S1. Crypto-слой + формат блоба.** Argon2id KDF, XChaCha20-Poly1305, тесты round-trip. Чисто клиентская библиотека, без сети.
- **S2. `postcat-sync` сервер.** axum + SQLite, эндпоинты `/auth`, `/push`, `/pull`; хранит только шифртекст; Docker + одиночный бинарь; конфиг URL в клиенте.
- **S3. Sync engine в клиенте.** Фоновый цикл push/pull, разрешение конфликтов LWW, статус-индикатор, тумблеры (история/секреты).
- **S4. Полировка:** восстановление на новом устройстве, смена пароля (re-encrypt), «выйти/забыть».

## Managed secrets / серверные переменные (enterprise)

Отдельная подсистема поверх sync-сервера: компания централизованно заводит
секреты, клиент ссылается на них по имени (`{{managed:prod_db_password}}`),
но **не видит значение**. Это НЕ E2E — plaintext секрета живёт на сервере
(в их периметре), поэтому это осознанно другой контур доверия, отдельный от
E2E-блобов персональных данных.

**Server-execute — опционально, политика на уровне секрета.** Это не общий
режим приложения, а свойство конкретного секрета, которое админ выбирает
осознанно. Дефолт — masked delivery (проще); server-execute включается точечно
для самых чувствительных секретов.

- Запрос **без** `{{managed:*}}` всегда выполняется **локально**, как сейчас
  (полный waterfall-тайминг, cookies, всё как раньше). Сервер не в цепочке.
- `masked delivery` (дефолт секрета): сервер отдаёт значение клиенту по TLS
  только на момент отправки; в UI маскировано, не персистится; запрос идёт
  с клиента. Слабее: в памяти клиента значение всё же есть → извлекаемо.
- `server-execute only` (опт-ин на секрет, строгая гарантия): клиент шлёт спек
  с плейсхолдером на сервер; сервер подставляет секрет и сам делает HTTP-вызов
  (переиспользуя `http_engine`), клиенту — только ответ. Секрет физически не
  покидает сервер. Плюс централизованная ротация/отзыв/аудит. Модель
  egress-прокси; сервер видит эти запрос+ответ (свой сервер).

**Что нужно на сервере для этого:**
- аккаунты + команда/организация, роли (admin / member);
- vault: секреты, сгруппированы; доступ per-группа или per-секрет;
- аудит-лог: кто/когда/какой секрет использовал в каком запросе;
- (для server-execute) серверный исполнитель запросов на нашем движке.

**Разделение контуров на сервере:**

```
 postcat-sync (self-host)
 ├─ blob store (E2E)     — только шифртекст персональных данных, сервер слеп
 └─ managed vault        — plaintext секретов компании, RBAC, аудит,
                           server-side execution для {{managed:*}}
```

## Осознанно НЕ делаем (пока)

- Командные воркспейсы, роли, комментарии, биллинг — это отдельный продукт и путь Postman к feature creep.
- Слияние (3-way merge) внутри приложения — Git-mirror закрывает это для тех, кому надо.
- Серверное чтение контента / governance-инспекция — противоречит E2E; если конкретная компания попросит server-readable режим, это отдельный флаг сервера, не дефолт.
